# Desktop Migration Draft Editing Design

## Context

桌面端已经具备知识迁移最小闭环：选择 `md/zip`，调用本地 `Ollama`，生成迁移预览，再写入知识库。当前最大缺口不是“能不能导入”，而是“用户无法修正模型结果”。

如果分类、标题或摘要有偏差，用户现在只能：

- 直接导入，再去文档页二次修改
- 或放弃本次迁移

这会让迁移工作流显得粗糙，也降低导入信心。

## Goal

让知识迁移预览成为“可控草稿”，用户可以在导入前做轻量修正，并在导入后直接跳到对应文档继续精修。

## Chosen Approach

在现有 `Assets` 页的 `知识迁移` 面板内，直接把只读预览升级为可编辑候选列表：

- 每条候选项支持编辑 `title / type / summary`
- 支持移除候选项
- 导入结果支持“打开文档”

不新增一级导航，也不在资源页放完整 Markdown 编辑器。

## Why This Approach

- 保持迁移工作台的紧凑性
- 让模型结果变成“草稿建议”而不是“最终裁决”
- 和已有文档编辑页形成清晰分工：迁移页做轻修，文档页做深修

## UX

### Draft List

每个迁移草稿卡展示并允许编辑：

- 标题
- 类型（`note / decision / review / reference`）
- 摘要
- 建议路径（只读）
- 来源标签（只读）
- 移除按钮

### Import Result

导入成功后结果面板继续展示：

- 已创建路径
- 每个路径旁增加“打开文档”按钮

点击后进入 `/documents/edit?path=...`。

## State Rules

- `migrationPreview.drafts` 变成前端可编辑状态，而不是只读快照
- 用户修改任意字段后，导入 payload 使用修改后的值
- 删除全部草稿后禁用“导入知识”
- 重新生成预览时，清空旧的导入结果

## Testing

新增前端测试覆盖：

- 可以在迁移预览中修改标题、类型、摘要
- 移除候选项后不会再出现在导入 payload 中
- 导入后点击“打开文档”会跳转到编辑页
