# Desktop Migration Preview Review Design

## Context

桌面端已经具备知识迁移的最小闭环：用户可在资源页选择 `md` 或 `zip`，调用本地 `Ollama` 生成候选草稿，再确认导入知识库。当前问题是迁移预览过于压缩，用户只能编辑 `标题 / 类型 / 摘要`，难以在导入前判断正文质量和原始来源是否合理。

## Goal

把知识迁移预览提升为可审阅的草稿队列，让用户在导入前看到更完整的候选信息，并在导入后直接进入文档编辑器继续精修。

## Approach Options

### Option 1: Keep Migration In Assets Page

继续将知识迁移放在资源页中，但增强预览结构：
- 列表展示候选项的核心信息
- 每条候选项支持展开查看原始来源与即将写入的 Markdown 正文
- 导入结果逐条提供“打开文档”

优点：
- 改动连续，复用现有状态和路由
- 不引入新的导航与路由复杂度

缺点：
- 资源页继续承载两类工作流，信息密度偏高

### Option 2: Split Migration Into Dedicated Route

将知识迁移抽成 `/assets/migrate` 独立工作区。

优点：
- 信息架构更纯粹
- 可进一步发展为导入向导

缺点：
- 需要同步调整导航、路由、测试和文档
- 超出这轮“增强现有闭环”的范围

### Option 3: Use Modal Review

预览列表保持简洁，点击候选项打开 modal 审阅正文与原文。

优点：
- 主界面更克制

缺点：
- 桌面端长文本审阅在 modal 中体验较差
- 连续编辑会增加来回切换成本

## Decision

采用 Option 1。继续使用资源页中的知识迁移工作台，但把预览区升级成可展开的草稿审阅列表。

## UX Design

### Migration Preview

- 预览列表保留现有的 `标题 / 类型 / 摘要` 编辑能力。
- 每条候选项新增展开/收起操作。
- 展开后展示两块只读信息：
  - `原始来源`：显示来源文件或压缩包条目标签
  - `正文预览`：显示即将导入的 Markdown 正文
- 保留单条移除操作，避免新增复杂批量流程。

### Import Result

- 导入结果区按路径展示已创建文档。
- 每个结果项提供 `打开文档` 按钮，直接进入 `/documents/edit?path=...`。
- 结果区保留 warning 展示，避免静默忽略问题。

## Data And State

- 前端为每个候选项维护本地展开状态，不修改后端返回结构。
- 导入结果沿用现有 `createdPaths`，前端为每个路径渲染独立跳转动作。
- 不新增后端命令，仍使用现有 `analyze_knowledge_migration` 和 `import_knowledge_migration`。

## Testing

- 在 `App.test.tsx` 增加迁移预览展开测试，确认用户可查看正文预览与来源信息。
- 增加导入结果逐条打开文档的测试，确认按钮会导航到文档编辑页并触发加载。
- 现有知识迁移测试继续保留，避免回归。
