# 文档对齐设计

## 目标

让仓库文档准确反映当前代码，包括当前运行行为、当前仓库结构、当前 CI / release 行为，以及 1.0 中尚未完成项的真实状态。

## 设计

文档分成两类：

1. 当前权威文档  
   直接描述当前仓库和当前 CLI 行为，必须与代码完全一致。
2. 历史规划文档  
   保留历史背景，但不能再误导读者理解当前路径、当前命令或当前实现边界。

当前权威文档包括：

- `README.md`
- `docs/requirements/knowledge-galaxy-1.0.md`
- `docs/specs/knowledge-galaxy-1.0-spec.md`
- `docs/specs/repository-layout.md`
- `docs/tasks/knowledge-galaxy-1.0-tasks.md`
- 各实现目录下的 README
- `skills/knowledge-galaxy-cli/*`

`docs/plans/` 下的历史文档将统一补充一个短状态头，说明：

- 它属于哪个历史阶段
- 当前应以哪些文档为准
- 文中的旧路径、旧命令是否只代表历史上下文

对于最容易误导人的历史文档，还要直接补充旧路径到当前路径的映射说明，例如 `scripts/kg`、`packages/rust` 等。

## 关键决策

- 代码是文档对齐的唯一事实来源。
- 1.0 中尚未实现的 asset import、export、reference / asset validation 必须继续写成未来工作，不能写成“当前已支持”。
- 历史文档保留历史身份，不伪装成“按当前代码编写的文档”。
- 新增一份面向未来的文档治理规范，明确后续每次代码变更时哪些文档必须同步。

## 验收标准

- 当前权威文档中不再出现“`--repo` 必填”这类过时结论。
- 当前权威文档不再把 Go / Rust 描述成仅用于构建的实现。
- 当前权威文档会明确区分“已实现功能”和“仍在计划中的功能”。
- 历史 plan 文档都有明显的历史状态头，不会继续被当成当前操作手册。
- 仓库中存在一份未来可执行的文档治理规范，并且当前文档已经链接到它。
