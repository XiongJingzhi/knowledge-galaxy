# Knowledge Galaxy 桌面端首页总览与 CI 分层设计

## 目标

把桌面端从“直接进入多工作区工作台”收敛成“首页总览 + 二级功能页”的结构，同时补上桌面端品牌 logo、将 GitHub Actions 按职责拆分为 `ci.yml` / `integration.yml` / `release.yml`，并把当前实现状态同步回文档。

## 当前问题

1. 首屏信息过载
当前桌面端一进入就堆叠仓库切换、总控台、统计卡、最近操作和工作台内容，主内容已经超过一屏，首页没有形成真正的入口页。

2. 导航层次不清
`文档 / 创建 / 资源 / 校验与导出 / 项目` 现在都像并列一级页，但用户第一次进入更需要的是全局概览、搜索和快速跳转。

3. 品牌资产缺失
桌面端当前没有独立的“知识星系”图形标识，视觉记忆点不足，也不利于在侧栏、README、后续窗口图标中复用。

4. CI 职责混杂
当前 `.github/workflows/ci.yml` 同时承担回归、构建、产物上传与 nightly 发布，阅读和维护成本都偏高。

5. 文档滞后
仓库根 README、桌面端 README 和历史 CI 设计文档还没有反映目录选择、结构总控台、桌面端测试构建接入等当前状态。

## 方案比较

### 方案 A：维持多一级工作区，仅压缩布局

优点：
- 改动小
- 风险低

缺点：
- 不能解决“首页不是首页”的问题
- 一级结构仍然偏工具台，不像控制台入口

### 方案 B：首页总览 + 二级功能页

优点：
- 与“一级页面做概要内容展示，二级承载功能”完全一致
- 能把桌面端首屏稳定控制在一屏内
- 有利于把 logo、全局搜索和快捷入口集中到品牌首页

缺点：
- 需要重构 `App.tsx` 导航和测试
- 需要为二级页补返回路径和上下文

### 方案 C：首页 + 右侧抽屉承载所有二级功能

优点：
- 保留单页面体验

缺点：
- 状态管理更复杂
- 不如真正的二级页清晰

## 采用方案

采用方案 B。

## 信息架构

### 一级页面

桌面端默认进入 `home`：

- 仓库选择
- 全局搜索
- 核心指标
- 最近操作摘要
- 快捷入口卡片

首页只负责“看全局、快速进入、回顾当前仓库状态”，不直接承载长表单和大块编辑器。

### 二级页面

保留五个功能页，但降为二级工作页：

- `documents`
- `create`
- `assets`
- `ops`
- `projects`

二级页顶部统一显示：

- 返回首页
- 当前区段标题
- 该页专属的快捷动作

## 首页行为

1. 全局搜索
- 首页搜索框支持直接输入
- 回车后跳转到 `documents`
- 同时带入搜索词

2. 快捷入口卡
- 每张卡只展示摘要和进入按钮
- 不在首页展开完整功能内容

3. 最近操作
- 首页只展示最近 3-5 条
- 更强调“回流摘要”，不做长日志面板

4. 一屏约束
- 首页桌面分辨率下控制在一屏
- 允许轻微滚动作为容错，但默认不应依赖滚动才能理解结构

## 视觉方向

首页采用“星系总控台”方向：

- 左侧品牌区加入结构化星系 logo
- 主区顶部是仓库与搜索
- 中部是核心指标和当前区段
- 下部是二级页入口卡
- 背景保持深色结构网格，但核心视觉锚点从大面板改为更接近轨道星系的图形组织

二级页则回到更高密度的工作台布局，保证功能效率。

## Logo 方案

logo 使用 SVG 生成，保持可读、可缩放和可复用：

- 中心核心点
- 多条轨道环
- 少量节点点位
- 不做写实插画

输出优先放在 `apps/desktop/src/assets/` 或 `apps/desktop/public/`，并在侧栏品牌区和 README 中复用。

## CI 拆分

### `ci.yml`

职责：
- 作为顶层入口
- 触发 PR / push 的基础验证
- 可调用 `integration.yml`

### `integration.yml`

职责：
- 仓库级测试
- desktop 前端测试
- desktop build
- desktop `cargo check`
- 其他实现的常规回归

这是开发过程中的主工作流。

### `release.yml`

职责：
- 构建 Go / Rust / desktop 产物
- 上传 artifact
- nightly / release 发布

发布逻辑不再和日常回归混在一个文件里。

## 测试策略

1. 前端
- `App.test.tsx` 覆盖首页默认渲染
- 覆盖首页搜索跳转到 `documents`
- 覆盖从首页进入二级页

2. 资源
- logo 文件存在并被侧栏使用

3. CI
- workflow YAML 至少通过语法层验证
- 本地无法完全执行 Actions，但应保证命令路径和依赖与仓库现状一致

## 不做的内容

- 不在本轮引入路由库
- 不做桌面端 installer / 签名 / notarization
- 不做复杂动画系统
- 不把二级页拆成大量新文件，优先在现有结构上收敛
